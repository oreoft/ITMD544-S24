Yifan's Answer to the Database Normalization Question

 

Initial Schema - Unnormalized (0NF)

CREATE TABLE Sales (
    OrderID INT PRIMARY KEY,
    CustomerName VARCHAR(255),
    CustomerEmail VARCHAR(255),
    BookTitle VARCHAR(255),
    AuthorName VARCHAR(255),
    AuthorCountry VARCHAR(255),
    BookGenre VARCHAR(255),
    PurchaseDate DATE,
    Quantity INT,
    PricePerUnit DECIMAL(10, 2),
    TotalPrice DECIMAL(10, 2)
);

Final SQL Schema - Normalized to 3NF

-- Customers Table
CREATE TABLE Customers (
    CustomerEmail VARCHAR(255) PRIMARY KEY,
    CustomerName VARCHAR(255)
);


-- Authors Table
CREATE TABLE Authors (
    AuthorName VARCHAR(255) PRIMARY KEY,
    AuthorCountry VARCHAR(255)
);


-- Books Table
CREATE TABLE Books (
    BookTitle VARCHAR(255) PRIMARY KEY,
    AuthorName VARCHAR(255),
    PricePerUnit DECIMAL(10, 2),
    FOREIGN KEY (AuthorName) REFERENCES Authors(AuthorName)
);


-- Orders Table
CREATE TABLE Orders (
    OrderID INT PRIMARY KEY,
    CustomerEmail VARCHAR(255),
    PurchaseDate DATE,
    FOREIGN KEY (CustomerEmail) REFERENCES Customers(CustomerEmail)
);


-- OrderDetails Table
CREATE TABLE OrderDetails (
    OrderID INT,
    BookTitle VARCHAR(255),
    Quantity INT,
    TotalPrice DECIMAL(10, 2),
    PRIMARY KEY (OrderID, BookTitle),
    FOREIGN KEY (OrderID) REFERENCES Orders(OrderID),
    FOREIGN KEY (BookTitle) REFERENCES Books(BookTitle)
);


-- BookGenres Table
CREATE TABLE BookGenres (
    BookTitle VARCHAR(255),
    Genre VARCHAR(255),
    PRIMARY KEY (BookTitle, Genre),
    FOREIGN KEY (BookTitle) REFERENCES Books(BookTitle)
);

Optimization theory

  1. 1NF: Ensures data atomicity and establishes a primary key for each table, removing repeating groups.

  2. 2NF: Removes partial dependencies, which ensures all attributes are dependent on the whole primary key, reducing redundancy.

  3. 3NF: Removes transitive dependencies, ensuring that attributes are only dependent on the primary key, not on other non-prime attributes, which reduces the chances of anomalies and increases data integrity.

Step 1: Identify Functional Dependencies

To start, we identify the functional dependencies in the unnormalized schema. A functional dependency, (X \rightarrow Y), means the attribute (Y) is functionally dependent on (X), where (X) can determine (Y).

  1. OrderID (\rightarrow) CustomerName, CustomerEmail, PurchaseDate (assuming each order uniquely identifies a customer and a purchase date)

  2. CustomerEmail (\rightarrow) CustomerName (assuming each email is linked to a single customer name)

  3. BookTitle (\rightarrow) AuthorName, AuthorCountry, BookGenre, PricePerUnit (assuming each book title uniquely identifies its author, country, genre, and price)

  4. AuthorName, AuthorCountry (\rightarrow) BookTitle, BookGenre (assuming an author from a specific country could write multiple books in various genres)

  5. OrderID, BookTitle (\rightarrow) Quantity, TotalPrice (assuming the quantity and total price are dependent on a specific order and book)

Step 2: Normalize to First Normal Form (1NF)

To convert to 1NF, we eliminate repeating groups and ensure each table has a primary key. Each attribute must contain atomic values only.

  1. Split the table into smaller tables where necessary to ensure atomicity.

  2. Ensure there's a primary key that uniquely identifies each row in a table.

  • Orders (OrderID, CustomerEmail, PurchaseDate)

  • Customers (CustomerEmail, CustomerName)

  • Books (BookTitle, AuthorName, PricePerUnit)

  • Authors (AuthorName, AuthorCountry)

  • OrderDetails (OrderID, BookTitle, Quantity, TotalPrice)

  • BookGenres (BookTitle, Genre) (Since a book can fall into more than one genre, we separate this into its own table)

Step 3: Normalize to Second Normal Form (2NF)

To achieve 2NF, we make sure that the schema is in 1NF and that all non-prime attributes are fully functionally dependent on the primary key.

  1. Remove partial dependencies, i.e., dependencies on part of a composite primary key.

  • Since OrderDetails depends on both OrderID and BookTitle for Quantity and TotalPrice, it's already in 2NF.

  • BookGenres is also in 2NF as its primary key (BookTitle, Genre) fully determines the genre of the book.

Step 4: Normalize to Third Normal Form (3NF)

To get to 3NF, we ensure the schema is in 2NF and that there are no transitive dependencies, where non-prime attributes depend on other non-prime attributes.

  1. Remove transitive dependencies.

  • In Books, AuthorName leads to a transitive dependency through AuthorCountry. This is already addressed in the 1NF step by separating Authors.

  • Ensure that attributes in Orders, Customers, Books, Authors, and OrderDetails depend only on their primary keys and not on other non-prime attributes.

Final Schema in 3NF

  • Orders (OrderID, CustomerEmail, PurchaseDate)

    • OrderID is the primary key.

  • Customers (CustomerEmail, CustomerName)

    • CustomerEmail is the primary key.

  • Books (BookTitle, AuthorName, PricePerUnit)

    • BookTitle is the primary key.

  • Authors (AuthorName, AuthorCountry)

    • AuthorName is the primary key.

  • OrderDetails (OrderID, BookTitle, Quantity, TotalPrice)

    • OrderID and BookTitle form the composite primary key.

  • BookGenres (BookTitle, Genre)

    • BookTitle and Genre form the composite primary key.

This normalized structure will improve data consistency, reduce redundancy, and make the database more efficient for queries, updates, and maintenance.
